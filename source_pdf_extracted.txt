--- PAGE 2 ---\n1. Описание проекта 
В рамках практического задания реализовано веб-приложение «Файловое 
хранилище» (вариант 1) на платформе .NET Framework 4.7.2 с использованием 
ASP.NET MVC 5. Приложение обеспечивает: 
 
- загрузку файлов на сервер с сохранением на диск и в базу данных SQL Server 
LocalDB; 
- скачивание файлов по уникальному идентификатору; 
- настройку максимального размера файла (10 МБ) и папки хранения через 
web.config; 
- ведение таблицы Files (Id, OriginalFileName, SavedFileName, ContentType, Size, 
UploadDate). 
 
Для ускорения разработки использовался AI-ассистент Cursor с моделью Codex 5.3 
(medium). Код генерировался промптами, затем выполнялся, коммитился в Git и 
валидировался по чек-листу.

5. Чек-лист тестирования приложения 
1. Запуск и базовый интерфейс 
• ☐ Проект открывается в Visual Studio без ошибок. 
• ☐ При запуске (F5) открывается домашняя страница. 
• ☐ В меню навигации присутствуют ссылки «Загрузить файл» и «Скачать файл 
(введите Id в адресе)». 
2. Загрузка файла 
• ☐ Страница загрузки содержит форму с полем выбора файла и кнопкой 
«Загрузить». 
• ☐ При выборе файла размером до 10 МБ появляется сообщение «Файл загружен. 
Id = X». 
• ☐ Файл сохраняется в App_Data/Uploads с уникальным именем (GUID). 
• ☐ В БД таблица Files получает корректную запись. 
• ☐ Если файл не выбран – сообщение «Выберите файл». 
• ☐ Если файл >10 МБ – сообщение «Файл слишком большой. Максимум 10 MB». 
3. Скачивание файла 
• ☐ По адресу /File/DownloadFile/X файл скачивается с исходным именем. 
• ☐ При несуществующем Id – 404 «Файл не найден». 
• ☐ Если файл удалён с диска – сообщение «Физический файл отсутствует». 
4. Конфигурация 
• ☐ В web.config заданы ключи UploadFolder, MaxFileSizeBytes. 
• ☐ Строка подключения DefaultConnection указывает на LocalDB. 
• ☐ maxRequestLength >= 30720. 
5. Git и репозиторий 
• ☐ Присутствует папка .git. 
• ☐ История коммитов отражает этапы разработки. 
• ☐ Репозиторий загружен на GitHub. 
• ☐ Нет вложенных Git-репозиториев (исправлено).

6. Аналитический этап (выводы) 
Лучшие практики работы с нейронными сетями (на примере Codex 5.3 
medium) 
• Детализация промпта – указание имён классов, версии фреймворка. 
• Разбивка на микрошаги – один промпт = одна задача. 
• Избегать команд, специфичных для среды (Enable-Migrations, dotnet add package). 
• Ручной контроль after генерации (using, пространства имён). 
• Фиксация в Git после каждого успешного промпта с указанием номера. 
• Использование чек-листа для валидации каждого шага.

Cursor (Codex 5.3 medium) ускоряет разработку и подходит для учебных проектов, но
