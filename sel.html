В качестве предметной области выбрано веб-приложение «Файловое хранилище», реализующее загрузку, скачивание, удаление и учет пользовательских файлов. В работе использованы материалы двух этапов практики: анализ второго этапа с фокусом на устранение дефектов и итоговый отчет по проекту. Это позволило сопоставить требования, реализацию и результаты валидации по чек-листам. 1. ОПИСАНИЕ ПРОЕКТА Проект представляет собой веб-приложение класса «файловое хранилище». Базовые функции: авторизация пользователя, загрузка файлов на сервер, хранение метаданных в базе данных, скачивание файлов по идентификатору, удаление и переименование объектов. По исходным материалам в работе фигурируют два технологических контура: реализация на ASP.NET MVC 5 .NET Framework 4.7.2, SQL Server LocalDB и улучшенная реализация на ASP.NET Core 9 MVC, Bootstrap, SQLite. В обоих случаях архитектура ориентирована на серверную обработку, хранение файлов в выделенной папке и ведение регистра метаданных в таблице Files/UploadedFile. Для ускорения разработки и подготовки исправлений применялся AI-ассистент Cursor с моделью Codex 5.3 в т.ч. профиль medium, что позволило быстро формировать итерации кода и проверять гипотезы по улучшению качества. 2. ЦЕЛИ И ЗАДАЧИ ПРАКТИКИ Цель практики: разработать и проанализировать качество веб-приложения файлового хранилища, а также оценить эффективность применения ИИ при решении инженерных задач. Основные задачи практики: — спроектировать структуру проекта и реализовать базовый функционал загрузки/скачивания файлов; — настроить хранение данных в БД и обеспечить связность модели метаданных файлов; — проверить корректность работы сценариев авторизации и разграничения доступа; — выявить и исправить уязвимости: отсутствие лимитов размера, проверок типа файла и контроля владельца; — обеспечить корректную конфигурацию репозитория исключение пользовательских данных из Git; — провести валидацию по чек-листу и сформировать итоговые выводы. 3. Функции: загрузка, скачивание, список файлов, удаление, простейшая авторизация. Использовать SQLite. Инструкцию по запуску из VS Code приложи. Коммит: feat: initial app with auth, upload, download, delete, sqlite Результат: Приложение запустилось с первой попытки. Основной функционал сразу заработал: можно было войти, загрузить файл, посмотреть список файлов, скачать и удалить файл. Это был хороший результат, потому что ИИ быстро сделал рабочий прототип. Но после проверки кода и тестирования стало понятно, что первая версия была сделана не полностью. Найденные проблемы Самые большие проблемы были связаны с безопасностью и правами доступа. В первой версии не было ограничений на размер или тип загружаемых файлов. В FilesController.Upload не было проверки file.Length по верхнему лимиту, а также не было белого списка типов или расширений файлов. Это означало, что любой авторизованный пользователь мог загрузить очень большой файл и быстро заполнить диск сервера. Также не было модели владения файлами. Модель UploadedFile не содержала владельца файла, а действия Download, Delete и Rename работали только по id. Если бы в системе было несколько пользователей, любой пользователь мог бы управлять чужими файлами. Ещё одна проблема была связана с git. В.gitignore игнорировалась только база данных App_Data/*.db, но не папка App_Data/Uploads/*. Из-за этого загруженные пользовательские файлы могли случайно попасть в репозиторий. Также в проекте не было автотестов. Не было тестового проекта и тестов для авторизации, загрузки, скачивания, удаления, переименования и негативных сценариев: неверный вход, слишком большой файл, несуществующий id, CSRF для POST-запросов. Код тоже был не очень удобен для чтения. Большая часть логики находилась в одном Program.cs, константы не были вынесены отдельно, а UI был минимальным. При этом тестирование и чтение кода, который создаёт ИИ, могут быть затруднены. Изменений много, они появляются быстро, и их нужно внимательно проверять. Но если находить ошибки и точно писать, что нужно исправить, ИИ без проблем дорабатывает проект и реально экономит время. В итоге: 1. ИИ-ассистент эффективен как инструмент ускорения рутинных этапов разработки, но требует постоянной верификации решений разработчиком. 2. Ключевые дефекты ранней версии проекта были связаны не с функциональной полнотой, а с качественными атрибутами: безопасностью, контролем доступа и инженерной дисциплиной сопровождения. 3. Пошаговая валидация по чек-листу и фиксация изменений в Git позволили обеспечить контролируемое повышение качества продукта.
